## 2025-02-28 - [Canvas Animation Loop Anti-Pattern]
**Learning:** In Next.js/React applications, starting a `requestAnimationFrame` loop on component mount that constantly runs (even when there's nothing to draw on the canvas, like in the previous `ClickSpark` implementation) causes massive CPU and battery drain. Next.js does not magically optimize this away.
**Action:** Always start canvas animations based on user interaction (like `onClick`) and explicitly stop the loop (`cancelAnimationFrame` or exit the recursion) when the animation finishes or there are no items left to render.
## 2025-04-12 - [React.memo Formik State Leak Anti-Pattern]
**Learning:** In highly interactive React apps (like multi-step quizzes using Formik), passing top-level Formik `errors` and `touched` objects to individual child components like `FormQuestion` creates a massive render-cascade bottleneck. Because Formik creates entirely new object references for `errors` and `touched` on every keystroke anywhere in the form, `React.memo` becomes useless. The children re-render constantly even if their specific fields haven't changed.
**Action:** When wrapping components in `React.memo()`, pass only specific primitive values (`error`, `isTouched`, `value`) targeted to that exact child. This allows `React.memo`'s shallow comparison to correctly skip re-renders for unaffected fields, dramatically reducing CPU overhead on user input.
## 2025-04-12 - [Synchronous DOM Queries in Render Cycle Anti-Pattern]
**Learning:** Using `document.getElementById()` inside a React component's render flow (or closely bound helper functions evaluated during render like `optionIsChecked`) completely bypasses React's virtual DOM reconciliation. Not only is it exceptionally slow because it forces layout/paint calculation, it causes state inconsistency and creates hidden dependencies that are hard to test.
**Action:** Always derive UI logic strictly from React props or state. Replace DOM queries with purely functional evaluations (e.g., `value === expectedValue`).
