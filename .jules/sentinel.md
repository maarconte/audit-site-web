## 2025-03-03 - [Mass Assignment / IDOR Risk via Object Destructuring and Firebase Admin SDK]
**Vulnerability:** A Firebase Cloud Function accepted arbitrary fields like `listIds` from a client request and blindly sent them to a third-party service (Brevo). This is a textbook Mass Assignment / Insecure Direct Object Reference (IDOR) flaw, enabling attackers to subscribe themselves to privileged or hidden mailing lists. Further, input variables lacked type and length validation, which could be exploited due to the `admin.firestore()` usage bypassing regular Firestore Rules validations on document creation.
**Learning:** Even when security rules are correctly specified in `firestore.rules`, the `admin` SDK in Cloud Functions runs with elevated privileges and bypasses those rules entirely. Additionally, directly extracting destructured arrays or objects from client inputs and mapping them to external APIs carries severe risks of Mass Assignment.
**Prevention:** Strictly type check and enforce bounds (e.g., `typeof email === 'string' && email.length < 255`) on all destructured values in Cloud Functions. Exclude non-user-controllable fields (like target lists or permission levels) from the expected payload and hardcode them securely on the server-side.
## 2025-03-03 - [NoSQL Injection / DoS Risk via Unvalidated Object Payload in Firebase Admin SDK]
**Vulnerability:** The `scores` object payload in the `submitForm` Cloud Function was passed directly to `admin.firestore().collection().add()` without any type checking, size limitation, or depth validation. Since `admin.firestore()` bypasses `firestore.rules`, an attacker could submit an arbitrarily deep or large JSON object, or include malicious keys (e.g. `__proto__` pollution or NoSQL operator injections depending on the backend parser), leading to potential Denial of Service (DoS) during document serialization or unintended database corruption.
**Learning:** Even optional fields like `scores` that are destructured from the request body must be explicitly validated before being passed to Firebase Admin SDK methods. Defaulting to `scores || {}` does not prevent a malicious object from being passed. Object inputs require validation that they are flat (not arrays or deeply nested objects), have bounded key counts and lengths, and contain the expected value types.
**Prevention:** For dynamic object inputs (like `Record<string, number>`), strictly validate using `typeof obj === "object" && obj !== null && !Array.isArray(obj)`. Limit the number of keys using `Object.keys().length`, limit the length of each key, and strictly type check every value (e.g., `typeof val === "number"`) before saving to Firestore.
