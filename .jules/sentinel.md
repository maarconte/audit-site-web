## 2025-03-03 - [Mass Assignment / IDOR Risk via Object Destructuring and Firebase Admin SDK]
**Vulnerability:** A Firebase Cloud Function accepted arbitrary fields like `listIds` from a client request and blindly sent them to a third-party service (Brevo). This is a textbook Mass Assignment / Insecure Direct Object Reference (IDOR) flaw, enabling attackers to subscribe themselves to privileged or hidden mailing lists. Further, input variables lacked type and length validation, which could be exploited due to the `admin.firestore()` usage bypassing regular Firestore Rules validations on document creation.
**Learning:** Even when security rules are correctly specified in `firestore.rules`, the `admin` SDK in Cloud Functions runs with elevated privileges and bypasses those rules entirely. Additionally, directly extracting destructured arrays or objects from client inputs and mapping them to external APIs carries severe risks of Mass Assignment.
**Prevention:** Strictly type check and enforce bounds (e.g., `typeof email === 'string' && email.length < 255`) on all destructured values in Cloud Functions. Exclude non-user-controllable fields (like target lists or permission levels) from the expected payload and hardcode them securely on the server-side.

## 2025-03-03 - [Strict Dynamic Payload Validation & Error Log Sanitization in Firebase]
**Vulnerability:** A Firebase Cloud Function accepted a dynamic `scores` object and inserted it directly into Firestore using `admin.firestore().collection().add()`. Because `admin` bypasses Firestore rules, unvalidated dynamic objects introduce severe Mass Assignment and NoSQL Injection risks. Furthermore, external API errors and generic exceptions were logged in their entirety, potentially leaking PII and server stack traces to logs.
**Learning:** Never trust dynamic objects (e.g., `Record<string, unknown>`) when using the Firebase Admin SDK. Even if the type is restricted in TypeScript, the runtime payload can be deeply nested or contain thousands of massive keys causing a DoS or injection. Also, external API error objects can contain the entire request body or user data within them.
**Prevention:** Strictly validate dynamic objects by ensuring they are flat objects (`typeof obj === 'object' && obj !== null && !Array.isArray(obj)`), limit the number of keys, enforce maximum key and value lengths, and restrict values to safe primitives. For logging, extract only known safe properties like `code` and `message`, and use `error instanceof Error ? error.message : String(error)` in catch blocks to prevent stack trace leakage.
