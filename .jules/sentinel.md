## 2025-03-03 - [Mass Assignment / IDOR Risk via Object Destructuring and Firebase Admin SDK]
**Vulnerability:** A Firebase Cloud Function accepted arbitrary fields like `listIds` from a client request and blindly sent them to a third-party service (Brevo). This is a textbook Mass Assignment / Insecure Direct Object Reference (IDOR) flaw, enabling attackers to subscribe themselves to privileged or hidden mailing lists. Further, input variables lacked type and length validation, which could be exploited due to the `admin.firestore()` usage bypassing regular Firestore Rules validations on document creation.
**Learning:** Even when security rules are correctly specified in `firestore.rules`, the `admin` SDK in Cloud Functions runs with elevated privileges and bypasses those rules entirely. Additionally, directly extracting destructured arrays or objects from client inputs and mapping them to external APIs carries severe risks of Mass Assignment.
**Prevention:** Strictly type check and enforce bounds (e.g., `typeof email === 'string' && email.length < 255`) on all destructured values in Cloud Functions. Exclude non-user-controllable fields (like target lists or permission levels) from the expected payload and hardcode them securely on the server-side.

## 2025-03-15 - [Mass Assignment / NoSQL Injection Risk via Unvalidated Nested Object]
**Vulnerability:** The `scores` object parameter in the `submitForm` Firebase Cloud Function was accepted from the client request payload without any type checking or validation. Since the parameter was passed directly to `admin.firestore().collection().add()`, which bypasses regular security rules, an attacker could potentially inject arbitrary key-value pairs or oversized datasets into the Firestore database, risking mass assignment and excessive database costs.
**Learning:** Even when the primary fields (like email and names) are strictly validated, all nested structures, objects, or arrays must also undergo rigorous structural and type validation before being stored with the elevated `admin` SDK in Firebase.
**Prevention:** Explicitly validate the shape of all objects submitted. Ensure they are actual objects (not null or arrays), enforce a strict maximum number of allowed keys, limit the key string length, and verify that the types of all nested values perfectly match the application's expected schema (e.g., only numbers for a scoring object).
