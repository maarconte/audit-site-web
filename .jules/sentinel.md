## 2025-03-03 - [Mass Assignment / IDOR Risk via Object Destructuring and Firebase Admin SDK]
**Vulnerability:** A Firebase Cloud Function accepted arbitrary fields like `listIds` from a client request and blindly sent them to a third-party service (Brevo). This is a textbook Mass Assignment / Insecure Direct Object Reference (IDOR) flaw, enabling attackers to subscribe themselves to privileged or hidden mailing lists. Further, input variables lacked type and length validation, which could be exploited due to the `admin.firestore()` usage bypassing regular Firestore Rules validations on document creation.
**Learning:** Even when security rules are correctly specified in `firestore.rules`, the `admin` SDK in Cloud Functions runs with elevated privileges and bypasses those rules entirely. Additionally, directly extracting destructured arrays or objects from client inputs and mapping them to external APIs carries severe risks of Mass Assignment.
**Prevention:** Strictly type check and enforce bounds (e.g., `typeof email === 'string' && email.length < 255`) on all destructured values in Cloud Functions. Exclude non-user-controllable fields (like target lists or permission levels) from the expected payload and hardcode them securely on the server-side.

## 2025-03-03 - [Overly Permissive CORS Configuration in Firebase Cloud Functions v2]
**Vulnerability:** A Firebase Cloud Function was configured with `cors: true` in its `onRequest` options. In Firebase Functions v2, setting `cors: true` allows all origins (`*`) by default, exposing the endpoint to potential Cross-Origin Resource Sharing attacks, CSRF, and data exfiltration from unauthorized domains.
**Learning:** Using `cors: true` is convenient for development but insecure for production. To implement secure and configurable CORS origins in Firebase Cloud Functions v2 using `defineString`, passing it to the `onRequest` options object does not work, as it throws an error during the Firebase CLI's function discovery phase. Instead, the `cors` package must be dynamically instantiated inside the function handler using `require('cors')({ origin: ... })`. The remainder of the HTTP handler logic must be placed inside the middleware's callback to properly handle preflight OPTIONS requests without hanging the execution.
**Prevention:** Always restrict CORS to explicitly allowed domains. In Firebase v2, read a parameterized list of allowed origins (e.g., via `defineString`) inside the request handler. Instantiate `cors` dynamically and execute it with the rest of the function logic inside its callback.
