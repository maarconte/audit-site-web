## 2025-03-03 - [Mass Assignment / IDOR Risk via Object Destructuring and Firebase Admin SDK]
**Vulnerability:** A Firebase Cloud Function accepted arbitrary fields like `listIds` from a client request and blindly sent them to a third-party service (Brevo). This is a textbook Mass Assignment / Insecure Direct Object Reference (IDOR) flaw, enabling attackers to subscribe themselves to privileged or hidden mailing lists. Further, input variables lacked type and length validation, which could be exploited due to the `admin.firestore()` usage bypassing regular Firestore Rules validations on document creation.
**Learning:** Even when security rules are correctly specified in `firestore.rules`, the `admin` SDK in Cloud Functions runs with elevated privileges and bypasses those rules entirely. Additionally, directly extracting destructured arrays or objects from client inputs and mapping them to external APIs carries severe risks of Mass Assignment.
**Prevention:** Strictly type check and enforce bounds (e.g., `typeof email === 'string' && email.length < 255`) on all destructured values in Cloud Functions. Exclude non-user-controllable fields (like target lists or permission levels) from the expected payload and hardcode them securely on the server-side.

## 2025-04-15 - [Overly Permissive CORS and Missing NoSQL Injection Validation via Admin SDK]
**Vulnerability:** The `submitForm` Firebase Cloud Function had overly permissive CORS (`cors: true`), allowing any domain to send cross-origin requests. Additionally, it lacked structural validation for the dynamic `scores` object before inserting it into Firestore. Because the `admin` SDK in Cloud Functions runs with elevated privileges and bypasses Firestore Rules completely, a malicious actor could pass massive objects (denial of service) or deeply nested/unexpected structures (NoSQL Injection/Mass Assignment).
**Learning:** Firebase Cloud Functions v2 can restrict CORS directly by passing an array of allowed origins to the `cors` property in `onRequest` options. More importantly, when inserting dynamic objects (like a `scores` dictionary) using the Admin SDK, one cannot rely on `firestore.rules` for protection. Explicit validation (checking for flat object structure, key limits, key length limits, and value types) must be enforced in the Cloud Function itself.
**Prevention:** Always hardcode allowed CORS origins for Cloud Functions (including `http://localhost:3000` for local dev). Always strictly validate the type, size, and content of all dynamic nested objects extracted from client requests before writing them to the database via the Admin SDK.
