## 2025-03-03 - [Mass Assignment / IDOR Risk via Object Destructuring and Firebase Admin SDK]
**Vulnerability:** A Firebase Cloud Function accepted arbitrary fields like `listIds` from a client request and blindly sent them to a third-party service (Brevo). This is a textbook Mass Assignment / Insecure Direct Object Reference (IDOR) flaw, enabling attackers to subscribe themselves to privileged or hidden mailing lists. Further, input variables lacked type and length validation, which could be exploited due to the `admin.firestore()` usage bypassing regular Firestore Rules validations on document creation.
**Learning:** Even when security rules are correctly specified in `firestore.rules`, the `admin` SDK in Cloud Functions runs with elevated privileges and bypasses those rules entirely. Additionally, directly extracting destructured arrays or objects from client inputs and mapping them to external APIs carries severe risks of Mass Assignment.
**Prevention:** Strictly type check and enforce bounds (e.g., `typeof email === 'string' && email.length < 255`) on all destructured values in Cloud Functions. Exclude non-user-controllable fields (like target lists or permission levels) from the expected payload and hardcode them securely on the server-side.

## 2025-03-03 - [NoSQL Injection via Unstructured Objects / Permissive CORS]
**Vulnerability:** A Cloud Function used an overly permissive CORS configuration (`cors: true`) which allowed any origin to call the endpoint. Additionally, the `scores` field, which is expected to be a `Record<string, number>`, lacked structural validation before being saved to Firestore. Since `admin.firestore()` bypasses Firestore Security Rules, this allowed potential Mass Assignment or NoSQL Injection vectors where an attacker could submit arbitrarily deep or large objects.
**Learning:** Setting `cors: true` in Firebase Cloud Functions v2 opens the function to all domains, which is risky for internal or domain-specific logic. For input validation, checking simple properties (like strings) is insufficient if complex objects (`Record<string, unknown>`) are accepted. Firestore rules don't protect data written via the Admin SDK.
**Prevention:** Always restrict `cors` to an array of specific origins (e.g., `['https://production.app', 'http://localhost:3000']`). For dynamic nested objects, explicitly validate that the input is a standard object (not an array or null), limit the maximum number of keys, enforce key length limits, and validate the type of every value before passing it to `admin.firestore()`.
